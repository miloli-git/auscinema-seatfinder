01

The product problem

Booking maps show availability, not quality.

Cinema seat maps tell you what's free. They don't tell you which free seat is actually worth sitting in.

Users want the good seat.

Centre, right distance from the screen, the right class (recliner / Gold / Vmax) — that judgement is left entirely to the user.

Don't replace checkout — route to it.

The app scores seats and hands off to the official booking path. No payment handling, no account, no replacing the cinema's flow.

02

Seven directions, three input types

Each direction is the same product on the same data (Toy Story 5 / Event George St / 6 sessions), so the only variable is design. They draw on three kinds of input.

1 · Functional baseline

Directions designed from first principles for the task, no external reference.

2 · Extracted design systems

Tokens (type, colour, motion, layout) pulled from live sites with Playwright, then diverged for product fit — reference-system extraction, not copying.

3 · Real auditorium data

Real captured seat geometry, classes and status, scored by the project's own model and rendered as a layout — not a synthetic grid.

03

Recommended direction: G — Glossier × SOH hybrid

  • 1Ranked rail — sessions sorted by best available seat, not just time.
  • 2Trust-building score — one number (98) the user can believe and act on.
  • 3Booking handoff — one clear path out to the official cinema checkout.
  • 4Real auditorium, glow heatmap — good seats literally light up; 156 real seats, class-driven widths.
  • 5Seat-class legend — standard / recliner / daybed read at a glance.

05

The UI is downstream of a real model

The recommended direction doesn't render a decorative grid. It renders a real captured auditorium — geometry, seat classes and status pulled through the project's own chain adapters and scored by its own model.

  • Distance to screen — rows too close are penalised; the sweet spot scores highest.
  • Centrality — horizontal offset from the centre line.
  • Row geometry — real col indices (Event cols decrease L→R), aisles and gaps respected.
  • Seat class — recliner / daybed / standard carry different widths and weight.
  • Status — sold / spacer / companion seats removed from the ranking.

Why it matters: it makes a backend / AI builder credible in design — the interface is the output of a system that actually behaves, not a theme.

06

What I learned, what I'd build next draft — make it yours

What held up

  • Design-system extraction gives useful constraints, but brand borrowing can't replace product fit.
  • Seat-map legibility beats aesthetic novelty — the prettiest direction wasn't the most usable.
  • The strongest direction is the one where scoring, trust and handoff are clearest, not the boldest one.

What's next

  • Take G into the React app and wire it to the live scorer (fixture scores are recomputed at render today).
  • Mobile booking path + accessibility pass on the seat map.
  • Reframe the entry point to be movie-forward — "see this film, well" — with chain / time / seat as downstream constraints.
  • User-test the score explanation with real moviegoers.
